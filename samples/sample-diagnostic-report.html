🔍 Why Are Tags Disappearing?

Root Cause: The Sentinel Incidents – Create Or Update API uses PUT (full-replace) semantics. When any service (including the Phishing Triage Agent) updates an incident, the entire incident object is replaced. If the labels array is omitted or empty in the PUT body, all existing labels are deleted.

The Defender XDR PATCH /api/incidents endpoint behaves similarly — it overwrites the tags array rather than merging. This is by-design API behavior, not a bug. The agent is not intentionally removing tags; it simply doesn’t preserve fields it didn’t set.

✅ Recommended Actions

Listed in order of effectiveness. Action 1 alone will resolve the immediate issue.

1
Deploy a Tag-Restoration Automation Rule (Immediate Fix)
Create an Automation Rule in Sentinel → Automation that triggers on Incident Updated and runs a Logic App / Playbook:
  1. GET the incident to read current labels.
  2. Compare against your required tags (e.g., AutoRemediate, PhishingReview).
  3. If any are missing, PUT the incident back with the full merged label set.
This auto-heals tags within seconds of removal. See the sample playbook below.
2
Re-prioritize Automation Rules
In Sentinel → Automation, give your tag-dependent rules a lower priority number (= higher priority) so they execute before the Phishing Triage Agent runs. This ensures your automation reads tags before they get stripped.
3
Move Automation Triggers to Immutable Properties
Tags are fragile because any PUT can overwrite them. More reliable trigger options:
  • Incident classification / determination — set by the agent, never cleared.
  • Alert product name — immutable after creation.
  • Incident comments — append-only, never overwritten.
4
Use Sentinel Watchlists for Automation Flags
Store automation flags in a Watchlist keyed by Incident ID. Playbooks query the watchlist instead of relying on tags. Tags can be informational only.
5
File Feedback with Microsoft
Request “protected tags” or merge-style tag semantics via the Defender Feedback Portal. This would prevent the issue at the platform level.

What the Playbook Does

Trigger:  Sentinel Automation Rule fires on "Incident Updated"
          ↓
Step 1:   GET /incidents/{id} — reads the full incident with current labels
          ↓
Step 2:   Compare current labels against your required tags list
          (e.g., AutoEscalate, Tier2-Review)
          ↓
Step 3:   IF any required tags are missing:
            - Merge: current labels ∪ missing tags
            - PUT /incidents/{id} with etag — writes the full label set back
            - Log: "Restored [missing tags]"
          ELSE:
            - Log: "Tags intact, no action needed"

Uses System Managed Identity (no credentials). The etag header prevents race conditions. Cost: < $1/month.

After Deployment — Verify It Works

  1. Open any phishing incident in Sentinel
  2. Manually remove a required tag (e.g., AutoRemediate)
  3. Wait ~30 seconds and refresh — the tag should reappear
  4. Check the Logic App Run History to confirm the playbook fired
